# FELHAVEN — Changelog

Dated history for the Felhaven dashboard (and the now-retired Metis voice
assistant), newest first.

**Entries are long on purpose.** Each records *why* a change was made and what it
cost — including the mistakes and what they taught — not just what changed. That
reasoning is the part worth keeping, and it lives here so the code and the other
docs don't have to carry it.

**Paths are written as they were at the time.** A file that has since moved or
been renamed keeps its old name in older entries; the entry that moved it says
so. A dated record that gets quietly edited to match the present isn't a record.

New here? Start with [`README.md`](README.md) for what Felhaven is,
[`SETUP.md`](SETUP.md) to run it, and [`CONVENTIONS.md`](CONVENTIONS.md) for how
the code is put together.

- `2026-07-17` — **Bug fix: Orpheus was decoding at 4× the RAM it needed to.** Caught in code review, one day after Orpheus shipped. `tools/orpheus.py` decoded every file at **48 kHz stereo** (`_SAMPLE_RATE = 48000`, `_CHANNELS = 2`) — matching the written handoff exactly, but the handoff itself was wrong: an amendment to 24 kHz mono had been given conversationally after the fact and never reached the doc, so the spec and the code agreed with each other and both disagreed with reality. Every `.opus` in `local_audio/` is produced by Echo at **24 kHz mono at origin** — kokoro synthesizes mono @ 24 kHz, and neither `calliope.save_wav()` nor Echo's ffmpeg encode step ever resamples or remixes — so asking ffmpeg to decode at 48 kHz stereo doesn't recover any information, it just upsamples and duplicates a channel for nothing. Measured on a real 6-minute-41-second file: **147 MB** decoded the old way vs. **37 MB** at its native rate — exactly 4×. Fixed to `_SAMPLE_RATE = 24000`, `_CHANNELS = 1`. The docstring is reframed to say what these constants actually are: Orpheus's own explicit *assumption* that `local_audio/` only holds Echo's output (a second, stated limit alongside the whole-file-into-RAM one) — not a generic decode default, and not a guard against non-Echo audio, which is deliberately left unenforced until the day it matters. `harmonia.py`'s docstring repeated the same wrong number in two places (it couldn't have caught the bug, since it just echoed the caller's claim) — corrected in lockstep. `_DECODE_TIMEOUT = 120.0` was checked against the real file live (0.8 s to decode 6m41s — ffmpeg decode is far faster than real time) and left as-is; it was never actually at risk, just picked against a smaller "3-minute briefing" example before any real file existed. `tests/test_orpheus.py` updated (`_decode`'s reshape test, the explicit-rate assertion in `play_file`). No behavior change to `harmonia.py`, `calliope.py`, or Echo — this was Orpheus's decode call only. Full suite **466 tests OK**, `mypy --strict --explicit-package-bases tools/` green.
- `2026-07-17` — **Orpheus's file list now shows each file's duration.** Requested from live use of the ORPHEUS tab — a bare filename didn't say how long a briefing was. `tools/orpheus.py` gained `_probe_duration(ffmpeg, path)`: rather than adding **ffprobe** as a second binary (it isn't bundled in `bin/` alongside `ffmpeg.exe`, and pulling in a whole extra binary for one number is a bad trade), it shells out to `ffmpeg -hide_banner -i <path>` with **no output** and reads the `Duration: HH:MM:SS.ss` line ffmpeg prints to stderr as part of its normal file-info banner — the standard ffprobe-free trick, and it never decodes a sample. That call always exits nonzero ("at least one output file must be specified"); expected, and ignored — only a missing Duration line counts as failure. Durations are **cached by filename** in a module-level dict (`_duration_cache`) and probed exactly once per file for the life of the process, not on every 2 s Kairos tick — `local_audio/` files are write-once (Echo never overwrites in place), so there's nothing to invalidate the cache on. A failed probe is deliberately **not** cached, so a transient issue (e.g. catching a file mid-write by Echo) can resolve itself on a later tick instead of sticking at "duration unknown" forever. `fetch()`'s `"files"` list changed shape from bare filenames to `{"name", "duration"}` rows — `list_files()` itself is untouched (still returns bare names; only the new `_file_rows()` wraps it), so nothing outside `fetch()`/the panel needed to change. `OrpheusPanel._reload_files()` now renders each file as a title row (click-to-play, unchanged) plus a small duration line underneath (`"m:ss"`, or `"duration unknown"` if ffmpeg couldn't read it — surfaced, not hidden). **Verified against the real `bin/ffmpeg.exe`** (not just the mocked tests): generated a 135 s silent Opus file with `ffmpeg`'s own `lavfi`/`anullsrc` and confirmed `_probe_duration` → `fetch()` → the panel's formatter round-trips end to end. - `2026-07-16` — **Harmonia — one owner for the audio output device — and Orpheus, the Vox Array's third tab.** After Echo, three things wanted to make sound (Calliope, a future Orpheus, and Morpheus) and nothing owned the device: Calliope's `_play_lock` protected only Calliope, and a lock across all three **can't work** — a Python mutex serializes *commands*, not *audio*, and mpv holds its own device handle in a separate process regardless of what any Python-side lock says. New app-root module **`harmonia.py`** (beside `kairos.py` — infrastructure, not a tool: no `fetch()`, no `handle()`) is now the *sole* caller of `sounddevice` in the process: `play(pcm, sample_rate, tag="")` yields Morpheus first (`morpheus.stop()`, one direction only — Morpheus never learns Harmonia exists) then enqueues; `stop()` is global (no channels: `sd.stop()` + drain the queue + bump an epoch); `is_playing()` is the read-only completion signal; `shutdown()` is wired into `felhaven._on_close` **before** `morpheus.shutdown()`. **Calliope was refactored onto it**: `_play`/`_play_lock`/the play-worker that called `sd.play()` directly are gone; the synth pipeline now hands each chunk to `harmonia.play()`, keeping the **prebuffer gate Calliope-side** (deciding *when* to release the first chunk of a turn, not *how* to drain a device queue — Harmonia must never learn what a filler is). **A deliberate, documented behavior change:** the old `stop()` only bumped an epoch (barge-in latency was one ~70-char chunk, unnoticeable); `harmonia.stop()` calls `sd.stop()`, which actually interrupts an in-flight `sd.wait()` — barge-in is now **instant**, and because Harmonia has no channels, a new question now also silences an in-progress Orpheus briefing (the accepted cost of one stream, one truth, not a bug). **`tools/orpheus.py`** + **`panels/orpheus_panel.py`** (the new **ORPHEUS** tab in `VoxArrayPanel`) play back one `.opus` file from `local_audio/` (Echo's output folder) — play/stop only, no pause/seek/playlists, "play what Echo produced" and nothing more. A whole-file ffmpeg decode into RAM (`-f f32le -ar 48000 -ac 2`, ~70 MB for a 3-minute briefing — a **stated limit**, not a bug; streaming would be a different module) is handed to `harmonia.play()` at the **explicit** 48 kHz Harmonia requires — kokoro is 24 kHz, and getting this wrong plays a briefing at half speed, the concrete reason `harmonia.play()`'s signature takes a sample rate at all (pinned by a dedicated test). Orpheus **is** a Kairos worker (2 s, unlike Echo) because its `fetch()` doubles as both the file-list refresh (no manual button — `local_audio/` is re-listed every tick) and the playback-finished signal the panel needs (`harmonia.is_playing()`, since `sd.wait()` blocks inside Harmonia's own thread and nothing else can observe completion); out of LLM scope for v1, same as Echo (no `handle()`).
